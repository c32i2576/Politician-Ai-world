# 文件撰寫規範

> 版本：1 ｜ 更新日期：2026-04-29 ｜ 狀態：current
> 適用對象：在 `docs/` 內新增或修改文件的所有人類與 AI
> 目的：讓未來的協作者（特別是 AI 助手）以一致方式維護文件，避免版號混亂、引用斷裂、舊檔殘留

新增或修改 `docs/` 內任何檔案前，先讀完本檔。違反本規範的 PR 應在合併前修正。

---

## 1. 目錄結構

```
docs/
├── 00-conventions.md       本檔（規範）
├── README.md               索引（必須與實際檔案保持同步）
├── NN-<topic>.md           頂層階段性文件（研究、缺口分析、決策紀錄等）
├── design/                 已釘下的子系統設計，每個子系統一份，僅 current 版本
│   └── <system>-model.md
└── archive/                被取代的歷史版本（只讀，不再修改）
    └── <原檔名>.md
```

### 1.1 何時放在哪一層

| 檔案性質 | 位置 | 範例 |
|---------|------|------|
| 跨子系統的研究、缺口、總覽、決策 | `docs/NN-*.md` | `01-research.md`, `02-gaps.md` |
| 單一子系統的具體設計（schema、公式、API） | `docs/design/<system>-model.md` | `bill-model.md`, `time-model.md` |
| 已被新版取代的舊檔 | `docs/archive/` | `design-priority-three.md` |
| 規範、流程、模板 | `docs/` 根目錄，`00-` 前綴 | 本檔 |

### 1.2 不要做的事

- ❌ 在檔名加版號後綴（`-v2`、`-v3`、`-final`、`-new`）
- ❌ 把多個子系統塞進一份檔（`design-priority-three.md` 是反面教材）
- ❌ 在 `archive/` 修改檔案內容（除了加歸檔頭、修死鏈）
- ❌ 跳過 `README.md` 更新就新增 / 重命名檔案

---

## 2. 命名規則

### 2.1 頂層檔：`NN-<topic>.md`

- `NN` 是兩位數閱讀順序，**不是版號**
- 數字代表「新人從 0 開始該照這個順序讀」
- 同一階段的多份檔可以共用一個區段（10-20 研究、20-30 缺口、30-40 決策…）
- 數字之間留空位，方便未來插入

### 2.2 子系統設計檔：`design/<system>-model.md`

- 一個子系統一份檔，檔名小寫連字號
- 不掛版號後綴；current 版本就是唯一檔
- 版號寫在**檔頭 blockquote**，不寫在檔名

### 2.3 歸檔檔：`archive/<原檔名>.md`

- 從 `docs/` 搬過去時保留原檔名（git mv 保留歷史）
- 即使原檔名有 `-v2` 之類後綴也不改名（保留原始識別）

---

## 3. 檔頭格式（強制）

每份 `.md` 檔的開頭必須是：

```markdown
# <標題>

> 版本：N ｜ 更新日期：YYYY-MM-DD ｜ 狀態：current | archived | superseded
> 對應缺口：[../02-gaps.md §M](../02-gaps.md)   ← 設計檔必填
> （其他可選 metadata：來源、取代、對齊…）
> 目的：<一句話說明本檔在解決什麼>

---

<正文開始>
```

### 3.1 三件套：版本 ｜ 更新日期 ｜ 狀態

- **版本**：純整數，從 1 開始。實質改寫才 +1（schema 重設計、演算法換掉、結構重排）。錯字、補例、refactor 不算
- **更新日期**：`YYYY-MM-DD`，每次實質改寫都更新
- **狀態**：
  - `current` — 當前有效版本
  - `archived` — 已被新檔取代，留作歷史參考
  - `superseded` — 邏輯上被取代但物理上還在原位（過渡期使用，最終應 → archived）

### 3.2 設計檔額外欄位

- **對應缺口**：必須指向 [`02-gaps.md`](02-gaps.md) 的某項。一份設計檔對應一個或多個缺口，這是 backlog 唯一索引
- **取代**：若取代舊檔，連結到 `../archive/` 內的舊檔
- **對齊**：若依賴另一份 design 檔的決定，連結並說明依賴關係
- **來源**：若內容是從別處抽出 / 整合，標明來源檔

## 4. 版號變更流程

### 4.1 小修（不動版號）

錯字、補例、調整文字順序、修死鏈：直接改，**不動版號與更新日期**。

### 4.2 實質改寫（版號 +1）

schema 重設計、公式換掉、章節重排、加新章節：

1. 改檔頭：版本 +1、更新日期改今天
2. 加 `## 0. 與 v(N-1) 的差異` 章節（用表格列關鍵變動）
3. 內文中若描述舊行為，明確標註「v(N-1) 寫…，v(N) 改為…」
4. **不要**把舊版搬走；新版直接覆蓋同一份檔

### 4.3 整份取代（產生新檔）

當變動大到「重命名子系統」「拆檔」「合檔」：

1. 在新位置建立新檔，版號從 1 開始（或延續，視情境）
2. `git mv` 舊檔到 `archive/`
3. 在歸檔檔的最頂端加歸檔頭：

   ```markdown
   > ⚠️ **此檔已歸檔** — 內容已遷移至：
   > - <章節> → [../<新位置>](../<新位置>)
   >
   > 保留作為歷史參考，**新工作請使用上述 current 版本**。
   ```

4. 把歸檔檔的 `狀態` 改為 `archived`
5. 在 `02-gaps.md` 對應缺口加上 `> **狀態：已解決** — 見 [新檔](新檔)` 標記
6. 更新 `README.md` 的目錄與快查表

## 5. 交叉引用規則

### 5.1 路徑

一律用相對路徑，依當前檔位置計算：

| 從 → 到 | 範例 |
|--------|------|
| `docs/A.md` → `docs/B.md` | `[B](B.md)` |
| `docs/A.md` → `docs/design/B.md` | `[B](design/B.md)` |
| `docs/design/A.md` → `docs/B.md` | `[B](../B.md)` |
| `docs/design/A.md` → `docs/design/B.md` | `[B](B.md)` |
| `docs/design/A.md` → `docs/archive/B.md` | `[B](../archive/B.md)` |

### 5.2 行號錨點

- 引用研究／缺口的具體段落用 `#L<行號>` 或 `#L<起>-L<迄>`
- **不要**引用 `design/` 檔的行號（design 檔常改，行號會漂移）。改用章節錨點 `#1-核心-schema` 或乾脆描述「見 §1.1」

### 5.3 缺口（gaps）是唯一 backlog 索引

- 設計檔的「對應缺口」一律指向 [`02-gaps.md`](02-gaps.md)
- 設計檔之間若有依賴，用 `對齊：` 欄位連結到對方檔，不要把對方檔當 backlog
- `02-gaps.md` 解決狀態必須與設計檔同步更新

---

## 6. README.md 維護

`README.md` 是索引，**任何檔案新增、重命名、歸檔都必須同步更新**：

1. 「文件地圖」樹狀圖
2. 「缺口解決狀態快查」表格（若涉及缺口）
3. 「閱讀順序」若有新頂層檔

更新 README 與檔案改動放在同一個 commit。

---

## 7. AI 助手特別注意事項

未來 AI 在這份專案內動文件時：

1. **先 grep 外部引用**：改檔名前用 `grep -r "<舊檔名>" docs/` 找出所有引用點，一併更新
2. **用 git mv 不用 mv**：保留歷史；若原檔未追蹤再用 `mv`
3. **批量改路徑用 sed**：`sed -i 's|<old>|<new>|g'`，但要先確認沒有歧義匹配
4. **不要主動建 v2 檔**：先檢查能不能直接改原檔（§4.2）；只有「整份取代」才走 §4.3
5. **歸檔的舊檔不要刪內容**：只加歸檔頭，正文保留作歷史參考
6. **README 與 02-gaps 是必動**：每次涉及檔案結構或缺口狀態的變動，這兩份要一起改
7. **檢查相對路徑**：寫 `../` 之前確認目前檔位置；寫完用 `grep -n "\.md\]" <檔>` 抽看所有連結
8. **commit 訊息**：用「動詞 + 對象」格式（`新增 bill-model 規範`、`歸檔 priority-three`），不要寫「更新檔案」這種空話

## 8. 範例：新增子系統設計檔

假設要新增「選民層」設計檔（對應 [02-gaps.md §4](02-gaps.md)）：

```bash
# 1. 建檔
docs/design/voter-model.md

# 2. 檔頭
> 版本：1 ｜ 更新日期：YYYY-MM-DD ｜ 狀態：current
> 對應缺口：[../02-gaps.md §4](../02-gaps.md)
> 對齊：[time-model.md](time-model.md)（民調聚合在 session close 觸發）
> 目的：定義選民 Agent 的行為模型、規模聚合方式、與議員的耦合介面

# 3. 更新 02-gaps.md §4
> **狀態：已解決** — 見 [design/voter-model.md](design/voter-model.md)（...一句話摘要...）

# 4. 更新 README.md
- 文件地圖加一行
- 快查表 #4 改為 ✅ 已解決

# 5. commit
新增 voter-model 設計檔（解決 gap #4）
```
